# FastAPI + Vue 3 + TypeScript + Vite + Konva/Fabric + Tauri 技术路线评估

本文档用于评估将当前项目的 UI 重心从 Qt 桌面端逐步转向“Web 前端 + 桌面壳”的可行性，重点说明这条路线的主要优点、主要难点，以及更适合当前仓库的落地方式。

这不是“立刻重写全部界面”的实施说明，而是一份偏架构决策与迁移路线的 roadmap 说明。

## 1. 背景

从当前仓库结构看，这个项目本质上已经是“同一核心引擎，多入口形态”：

- `manga_translator/` 提供检测、OCR、翻译、修复、排版等核心处理链
- `manga_translator/server/` 已提供 FastAPI 服务端与 Web 端静态资源
- `desktop_qt_ui/` 提供 Qt 桌面 UI 与可视化编辑器

这意味着：

1. 真正最有价值、最重的业务逻辑已经在 Python 核心层
2. UI 层并不是唯一入口，仓库本身已经具备 Web 化基础
3. 如果要重构 UI，最佳方向通常不是推倒 Python 核心，而是围绕现有核心能力重组交互层

因此，从仓库现状出发，`FastAPI + Vue 3 + TypeScript + Vite + Konva/Fabric + Tauri` 是一条比较自然的技术路线。

## 2. 这套技术栈各自负责什么

### 2.1 FastAPI

建议继续作为后端主入口，负责：

- 文件上传、任务创建、任务查询、结果下载
- 批量翻译、导出原文、导入翻译并渲染等工作流编排
- 配置读写、权限控制、审计、历史记录
- WebSocket / SSE 进度推送
- 调用现有 `manga_translator/` 核心处理链
- 提供给前端编辑器的 JSON、蒙版、修复图、导出结果等接口

对当前项目来说，FastAPI 不是新引入的方向，而是现有能力的继续强化。

### 2.2 Vue 3

建议作为主前端框架，负责：

- 主界面、设置面板、任务列表、结果页、管理页
- 编辑器外围 UI：工具栏、图层列表、属性面板、历史记录、快捷键提示
- 将翻译流程、编辑流程、资源管理流程组织成状态清晰的页面

Vue 3 的优势在于组件边界清晰、上手成本适中、对复杂表单和中后台界面友好。

### 2.3 TypeScript

建议作为前端的默认语言，负责：

- 约束前后端数据结构
- 明确文本框、区域、蒙版、任务状态、导出选项等数据模型
- 降低大规模 UI 重构时的隐性接口错误
- 为编辑器的复杂交互提供可维护的类型边界

对于本项目这类“编辑器 + 多种工作流 + 多状态任务”的系统，TypeScript 的收益不是锦上添花，而是长期维护成本控制的关键。

### 2.4 Vite

建议作为前端开发与构建工具，负责：

- 本地热更新与快速启动
- 独立管理 Web 前端构建产物
- 在开发态与 FastAPI 分离，在发布态再整合静态资源

它的核心价值不是“功能更多”，而是让前端开发体验和构建速度明显优于传统静态资源手工拼装方式。

### 2.5 Konva / Fabric

建议作为浏览器端编辑器画布层，负责：

- 图片显示、缩放、平移
- 文本框/区域框的选中、拖拽、旋转、缩放、变形
- 图层级交互与命中测试
- 画笔/橡皮擦等交互工具的前端表现
- 将用户操作序列化为可回放的编辑状态

它们的定位不是替代后端渲染，而是承接当前 Qt 编辑器在前端的交互职责。

### 2.6 Tauri

建议在需要桌面打包时作为外层桌面壳，负责：

- 把 Web UI 封装成桌面应用
- 提供本地文件系统、窗口、菜单、对话框等桌面能力
- 协助启动、管理和通信 Python 后端进程
- 生成 Windows / macOS / Linux 桌面分发包

Tauri 的核心价值不是“让 Web 看起来像桌面”，而是让项目能用一套 Web UI 同时覆盖浏览器版和桌面版。

## 3. 这条路线的主要优点

### 3.1 最大化复用现有 Python 核心

这是这条路线最大的优点。

当前项目最复杂、最有积累的部分在：

- 文本检测
- OCR 调度
- 翻译器接入
- 修复与上色
- 渲染与导出
- 工作流编排

这些都已经在 Python 层存在，而且很多逻辑并不依赖 Qt 本身。换句话说，真正应该被复用的是 `manga_translator/`，而不是把桌面 UI 当成唯一主干。

如果采用这条路线：

- 后端核心逻辑可以继续使用现有模块
- 迁移重点放在 UI 层，而不是模型能力重写
- 风险集中在交互迁移，而不是算法/模型迁移
- 能避免“功能没变、却因为技术栈替换而重写一遍核心流程”的无谓成本

### 3.2 更适合长期维护的分层结构

Qt 版本里，虽然服务层和核心层已经有一定分离，但桌面端依然承担了大量：

- 状态组织
- 交互逻辑
- 编辑器操作编排
- 局部配置处理
- UI 与业务耦合的桥接代码

迁到 Web 栈后，分层会更自然：

- FastAPI：任务、资源、权限、流程编排
- Vue + TypeScript：页面状态、组件树、交互逻辑
- Konva/Fabric：画布交互
- Tauri：桌面壳与系统能力

这种分层对后续扩展更友好，尤其适合：

- 同时维护 Web 版和桌面版
- 拆分编辑器与管理后台
- 引入更多前端工程化能力
- 让不同开发者在更明确的边界上协作

### 3.3 开发环境更稳，尤其适合 Windows / WSL 场景

当前 Qt 桌面端在 Linux、WSL、Wayland、X11、OpenCV Qt 插件混用、系统运行库缺失等场景下，都更容易遇到环境问题。

相比之下，Web 路线开发时：

- 浏览器就是主要渲染环境
- 前端和后端可以分别调试
- 不强依赖本地图形栈是否完整
- 在 Windows、WSL、远程环境下通常更稳定

对于开发者来说，这会直接降低：

- 首次进入项目的环境门槛
- UI 调试被系统库卡住的概率
- 跨平台联调成本

这并不意味着浏览器没有兼容性问题，而是这些问题通常比桌面 GUI 运行库问题更可控、更可观察。

### 3.4 同一套 UI 可以同时服务 Web 与桌面

这是相对纯 Qt 的另一个明显优势。

如果 Web 前端本身已经能完成：

- 配置编辑
- 翻译任务管理
- 图片结果预览
- 编辑器操作

那么桌面版不一定需要一套独立的 UI 实现。只要加上 Tauri：

- 浏览器版可以服务远程部署、局域网部署、多用户场景
- 桌面版可以服务本地单机用户
- 绝大部分 UI、交互、状态管理代码都可以共享

这比长期双维护“Qt UI 一套、Web UI 一套”更有规模效益。

### 3.5 前端生态更适合做复杂交互与可视化管理界面

当前项目不只是“翻译一张图”，它还有：

- 多工作流模式
- 多阶段状态展示
- 配置项分组与动态显示
- 用户资源管理
- 管理后台
- 编辑器侧栏、工具栏、属性栏、图层、历史

这些场景本质上更像现代 Web 应用和图形编辑器，而不是传统桌面表单软件。

前端生态在这些方面通常更强：

- 状态管理更成熟
- 表单、表格、树、抽屉、弹窗、路由、权限方案丰富
- 浏览器调试工具更成熟
- 视觉组件库选择更多
- 后续做文档、埋点、遥测、崩溃回溯也更方便

### 3.6 TypeScript 能显著降低迁移期的“隐形错配”

当前仓库跨层传递的数据并不少，例如：

- 文本框 geometry
- OCR 结果
- 翻译结果
- 修复图路径
- 导出选项
- 工作流模式
- 配置模型
- 任务状态

一旦进入前后端分离的形态，如果接口模型不清晰，就很容易出现：

- 后端字段改了，前端静默失效
- 某些流程只在部分模式下崩
- 编辑器里能打开，导出时却失败
- 同一数据在不同页面解释不一致

TypeScript 的价值就在于让这种错配尽可能前移到开发期暴露，而不是等用户点到某个按钮才发现。

### 3.7 更容易把“编辑器外”的能力做完整

Qt 编辑器很强，但一个完整产品不只有编辑器。

这条路线特别适合补齐这些能力：

- 任务中心
- 历史记录
- 用户资源管理
- Web 管理后台
- 配置模板导入导出
- 多设备访问
- 账户与权限体系
- 远程服务部署

这些能力中，很多已经在 `manga_translator/server/` 里有基础设施；Web 路线能更自然地把它们做成统一产品，而不是独立拼接的多个入口。

## 4. 这条路线最难的地方

### 4.1 最大难点不是“页面重写”，而是“编辑器重建”

如果只是把：

- 设置页
- 任务页
- 结果页
- 管理页

迁到 Vue，难度是可控的。

真正高难度的部分在于把 Qt 编辑器迁到浏览器端。当前 Qt 编辑器承担的能力包括但不限于：

- 文本框移动、旋转、变形
- 图元选择、命中测试、层级关系
- 画笔与橡皮擦
- 撤销/重做
- 图片切换与状态保持
- 导出前后的编辑状态同步
- 与后端 OCR / 修复 / 渲染结果联动

这不是“换个组件库”就能解决的事，而是需要重新设计编辑器的前端交互模型。

换句话说：

- Web 化容易的是外围页面
- Web 化最难的是画布与编辑器

### 4.2 浏览器画布与后端最终渲染之间容易产生“所见非所得”差异

当前项目里，真正生成最终结果的很大一部分逻辑仍在 Python 后端，包括：

- 字体布局
- 自动缩放
- 竖排/横排
- 气泡内对齐
- 智能断句后的排版
- 修复图与渲染图合成

但浏览器编辑器为了给用户即时反馈，也必须自己渲染一个预览层。

这会带来一个经典问题：

- 前端预览看起来是这样的
- 后端最终导出结果却有细微偏差

偏差可能来自：

- 不同平台字体度量不同
- Canvas/Text 渲染和 PIL/Qt 字体布局不同
- 竖排文本、换行规则、基线对齐算法不同
- 前端预览缩放与后端实际像素输出不完全一致

对漫画嵌字场景来说，这不是小问题，因为用户非常容易察觉文字位置、行距、气泡边距的偏移。

因此，这条路线必须尽早决定：

1. 浏览器端预览做到多高精度
2. 后端最终渲染是不是唯一真源
3. 允许多大的预览/导出偏差

如果这个问题不先设计清楚，编辑器后面会出现大量“看起来没问题、导出后不一样”的反馈。

### 4.3 大图、蒙版与高频交互对浏览器性能要求很高

本项目处理的是漫画图片，不是普通表单页面。常见场景包括：

- 高分辨率图片
- 多文本框
- 蒙版编辑
- 缩放平移
- 多图切换
- 结果预览与原图对照

在浏览器端，性能瓶颈通常集中在：

- 大图解码与显示内存
- Canvas 重绘成本
- 画笔轨迹频繁更新
- 多图层叠加
- 历史记录快照体积
- Python 与前端之间传输大尺寸图像或蒙版数据

Qt 的图元体系和桌面内存模型，与浏览器 Canvas/WebGL 的优化方式并不一样。迁移时不能简单假设“桌面端能做，浏览器端就自然能做”。

通常需要额外设计：

- 分层渲染策略
- 局部刷新
- 大图降采样预览
- 原图/修复图/蒙版分辨率分离
- 历史记录压缩与增量存储
- 前端只保存操作指令，最终结果交给后端重放

### 4.4 Konva 和 Fabric 都不是 Qt 编辑器的直接等价物

这两者都能做画布交互，但都不是“拿来就能完整替代 QGraphicsScene”的方案。

#### Konva 的优点

- 场景树和事件模型较清晰
- 性能通常比较稳定
- 适合自己掌控渲染层次和交互边界
- 与现代前端组件化集成通常更自然

#### Konva 的难点

- 文本编辑、复杂对象模型、富交互控件很多都需要自己补
- 如果编辑器需求继续增长，自定义工作量会比较高
- 更偏“你自己搭一套编辑器框架”

#### Fabric 的优点

- 自带对象模型更完整
- 选择、变换、序列化、对象级操作更现成
- 做“图形编辑器 MVP”通常更快

#### Fabric 的难点

- 一旦需求走到较深的自定义文本排版、复杂蒙版、强约束几何编辑，二次改造成本会升高
- 某些性能与事件控制场景下，可能需要绕开默认机制
- 对复杂场景的维护体验未必始终优于自己掌控的场景树方案

#### 实际判断

对于这个项目，困难不在于“选哪个库功能更多”，而在于：

- 文字框是不是矩形就够，还是要支持更复杂形状
- 蒙版编辑是轻量修补，还是接近专业画笔工具
- 预览文本是否要求高度贴近最终输出
- 历史记录是对象级撤销，还是命令级撤销

也就是说：

- 如果优先追求编辑器尽快形成 MVP，Fabric 往往更快
- 如果优先追求长期可控的定制化编辑器内核，Konva 往往更稳

但无论选谁，都仍然需要较大规模的自定义逻辑。

### 4.5 Tauri 不会自动消除 Python 打包复杂度

很多人看到 Tauri，会以为“桌面打包就轻了”。

这句话只对前端壳成立，不对 Python 核心成立。

当前项目真正重的部分并不在 UI，而在：

- Python 解释器
- torch / OCR / 推理依赖
- 模型文件
- 字体、词典、模板资源
- 更新逻辑与路径管理

所以即便采用 Tauri：

- Python 运行环境依然要打包或引导安装
- 模型资源依然要管理下载、缓存、升级
- 不同平台路径与权限差异依然要处理
- 后端进程的启动、关闭、日志、崩溃恢复依然要做

Tauri 能降低的是“桌面壳”的重量，不能自动解决“AI 应用分发”的重量。

### 4.6 桌面能力从 Qt 迁到 Tauri 时，会遇到新的系统边界

Qt 桌面端很多本地能力是天然的；换到 Tauri 后，很多能力都需要重新明确边界：

- 打开本地文件/文件夹
- 保存导出结果
- 拖拽目录
- 剪贴板
- 字体发现与加载
- 系统菜单和快捷键
- 多窗口
- 崩溃恢复
- 自动更新

这些能力不是做不到，而是要重新以“Tauri 能力 + Web 前端 + Python 后端”三层方式设计。

尤其要注意：

- 哪些动作必须由 Tauri 调系统 API
- 哪些动作应该由前端发起、后端执行
- 哪些动作只适合本地桌面，不适合纯浏览器版

如果不做边界梳理，后面很容易出现“浏览器版和桌面版行为不一致”的问题。

### 4.7 迁移期的双端维护成本不低

在相当长一段时间里，项目很可能会同时存在：

- 旧 Qt UI
- 新 Web UI
- 后端 Python 核心

如果迁移节奏控制不好，会出现：

- 同一配置项需要维护两份 UI 接线
- 同一工作流需要兼容两套前端入口
- 新功能不知道该先做 Qt 还是先做 Web
- Bug 修复需要两边同步验证

因此，这条路线不能按“所有端一起平推”的方式推进，而要明确：

1. 哪些能力先以 Web 为主
2. 哪些能力短期继续由 Qt 承担
3. 在什么节点后停止为 Qt 增加新能力

### 4.8 现有前端资源未必能直接平滑升级到现代 Vue 工程

当前仓库已经有 `manga_translator/server/static/`，但如果要真正切到 Vue 3 + TypeScript + Vite，通常意味着：

- 前端将从“静态资源页面”升级为独立工程
- 开发态可能需要前后端分离启动
- 发布态再把构建产物回收进服务端静态目录
- API、路由、鉴权、资源路径、缓存策略都要更清晰地定义

这不是障碍，但需要接受一个事实：

- 这是前端工程化升级
- 不是只往现有 `static/` 目录里多塞几个 JS 文件

## 5. 这条路线特别适合当前仓库的原因

### 5.1 核心引擎已经天然适合服务化

当前核心能力大多已经在 Python 模块里，而不是深嵌在 Qt 控件生命周期里。这意味着把它们暴露为服务接口相对自然。

对当前项目来说，最值得保留和强化的是：

- 核心流水线
- 任务编排
- 配置模型
- JSON 中间格式
- 工作目录规则

而不是继续把更多产品能力绑定到特定 GUI 框架。

### 5.2 仓库已经有 Web 基础，不是从零开始

当前并不是“纯桌面项目准备第一次做 Web”，而是已经有：

- `web` 模式入口
- `server/` 目录
- 管理后台与静态资源
- 账户、权限、历史、资源等服务端基础设施

这意味着这条路线不是另起炉灶，而是沿已有方向继续深化。

### 5.3 Web 端更容易承接未来的多人、多设备、远程部署需求

如果未来项目希望支持：

- 局域网访问
- 多用户配置隔离
- 管理后台
- 远程运行模型服务
- 统一资源管理

那么 Web 路线天然比纯 Qt 路线更顺。

即便最终仍保留桌面版，桌面版也可以变成 Tauri 包装的“本地单机壳”，而不是唯一交互入口。

## 6. 推荐的落地策略

不建议用“大爆炸式重写”。更适合当前仓库的是分阶段迁移。

### 阶段 0：先稳住后端边界

目标：先让 Python 核心与前端边界更清楚。

建议优先做：

- 梳理统一任务模型、配置模型、文本框 JSON 模型
- 明确哪些字段是前后端共享契约
- 把 Qt 私有状态与通用业务状态拆开
- 优先补可被前端直接调用的服务接口

这一阶段的关键，不是做漂亮页面，而是给后面迁移打“接口地基”。

### 阶段 1：先迁外围页面，不先碰完整编辑器

目标：先把最容易形成产出的部分迁走。

建议先迁：

- 首页/任务页
- 设置页
- 历史记录
- 管理后台
- 结果预览与下载
- 批量工作流入口

这样可以尽快获得：

- 一套可用的现代 Web UI
- 稳定的 API 与状态管理模式
- 对前后端边界的实际验证

### 阶段 2：实现编辑器 MVP

目标：先做“够用”的浏览器编辑器，而不是一步追平 Qt 全量能力。

建议 MVP 只覆盖：

- 图片显示、缩放、平移
- 文本框选中、移动、旋转、缩放
- 文本内容编辑
- 基础属性编辑
- 简单蒙版画笔
- 保存 JSON
- 调用后端导出预览/结果

这一步的关键是建立可用编辑闭环，而不是先做所有高级工具。

### 阶段 3：补高级编辑能力

在 MVP 跑通后，再逐步补：

- 更复杂的形状编辑
- 更强的画笔/橡皮擦能力
- 历史记录优化
- 多图切换状态保持
- 性能优化
- 更高精度的前后端渲染一致性

### 阶段 4：再考虑 Tauri 作为正式桌面壳

等 Web UI 和编辑器已经相对稳定后，再把它们封装为桌面版。

这样好处是：

- 不会让“桌面壳问题”掩盖真正的编辑器问题
- 可以先验证 Web UI 是否足够成熟
- Tauri 的职责更清晰：只负责桌面能力与打包，而不是承担产品重构本身

## 7. 对 Konva / Fabric 的具体建议

如果要给一个更偏当前仓库的建议：

### 方案 A：优先 Fabric

适合前提：

- 希望尽快做出编辑器 MVP
- 短期更重视对象编辑能力而不是长期自定义内核
- 目标是尽快把 Qt 编辑器的核心操作搬到浏览器里

优点：

- 起步快
- 现成对象操作能力较多
- 更容易较早形成“可用编辑器”

缺点：

- 后期定制度继续上升时，可能需要较多绕开默认机制的处理

### 方案 B：优先 Konva

适合前提：

- 预期编辑器会长期演进
- 更重视性能、场景树控制权和交互可塑性
- 愿意前期多写一些编辑器基础设施

优点：

- 长期更可控
- 更适合做“自定义编辑器内核”

缺点：

- 前期开发量更高
- 很多对象级能力需要自己搭

### 当前更现实的判断

如果目标是“尽快验证 Web 编辑器是否能替代 Qt 编辑器的核心路径”，通常更建议先偏向 **Fabric 路线做 MVP**；如果目标已经明确是“长期构建一个高度可定制、性能可控的漫画编辑器”，则 **Konva 路线更值得认真评估**。

但无论哪条路线，都建议先做小范围原型验证：

- 1 张大图
- 50~100 个文本框
- 基础移动/旋转/缩放
- 简单文本编辑
- 简单蒙版涂抹
- 导出结果回看

先用真实负载验证，再决定库选型，而不是只看 Demo 功能列表。

## 8. 最终结论

对当前仓库来说，`FastAPI + Vue 3 + TypeScript + Vite + Konva/Fabric + Tauri` 是一条很有吸引力、也相对务实的路线。

它最大的优点是：

- 能最大化复用现有 Python 核心
- 更适合长期维护和扩展
- 开发环境通常比 Qt 桌面端更稳定
- 可以统一 Web 版与桌面版 UI
- 更有利于承接管理后台、远程部署和多用户场景

它最大的难点是：

- 真正困难的是浏览器端编辑器重建
- 前后端渲染一致性很难天然保证
- 大图、蒙版和高频交互会带来明显性能挑战
- Tauri 并不会自动消除 Python 分发复杂度
- 迁移期间的双端维护成本需要严格控制

因此，更推荐的策略不是“立刻替换 Qt”，而是：

1. 保留 Python 核心不动
2. 先把 Web 前端工程化做起来
3. 先迁外围页面，再做编辑器 MVP
4. 等 Web 编辑器足够成熟后，再用 Tauri 封装正式桌面版

如果迁移顺序把握得当，这条路线很可能比继续深度押注单一桌面 GUI 框架更适合这个项目的长期演进。
